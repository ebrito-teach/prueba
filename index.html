📖Micro-lección — Antes de empezar

⚡ Lo que necesitas saber

Las bases de datos SQL almacenan información crítica: usuarios, contraseñas, datos financieros. Un atacante que logre ejecutar SQL malicioso puede leer, modificar o borrar toda esa información. Los tres pilares de una BD segura son: validación de entradas (nunca confiar en lo que el usuario envía), principio de mínimo privilegio (cada usuario solo hace lo que necesita), y autenticación robusta (contraseñas hasheadas, nunca en texto plano).

Estos tres principios son la diferencia entre una app segura y un titular que empieza con "Empresa filtra datos de 50,000 clientes..."

🚨Situación — Día 1 en DataTech RD

Tu jefe Marcos te llama a su oficina. "Bienvenido. Mira, no te voy a mentir: la semana pasada alguien entró a nuestra base de datos y robó 2,000 registros de clientes. El cliente nos está demandando." Marcos abre el código del sistema de login. "¿Ves el problema?"

👆 Haz clic en la línea que tiene el problema de seguridad:

def login_usuario(username, password): conn = get_db_connection() cursor = conn.cursor() query = "SELECT * FROM users WHERE user='" + username + "' AND pass='" + password + "'" cursor.execute(query) result = cursor.fetchone() return result is not None
Intentos:
consola — simulación de ataque

💡 Insight — SQL Injection

La línea 4 concatena strings directamente en la query. Un atacante escribe admin'-- como username y la query se convierte en SELECT * FROM users WHERE user='admin'--' AND pass='...'. El -- comenta el resto del SQL, ¡y el atacante entra como admin sin saber la contraseña!

✅ Identificaste la vulnerabilidad. Marcos asiente: "Correcto. Ahora necesito que lo arregles. Tienes 10 minutos."
✏️Decisión — Arregla el código

conn = get_db_connection()
cursor = conn.cursor()
query = "SELECT * FROM users WHERE user= AND pass="
cursor.execute(query, )
return cursor.fetchone() is not None

💡 Pista: los valores se reemplazan con ? y se pasan como tupla al execute()

Intentos:

💡 Insight — Queries Parametrizadas

Con cursor.execute(query, (username, password)), el driver SQL separa el código SQL de los datos. No importa qué escriba el usuario: siempre se trata como dato, nunca como código. Esto elimina completamente el SQL injection.

✅ Código corregido. Marcos lo revisa y levanta una ceja: "Buen trabajo. Pero mira los permisos del usuario de la base de datos..."
🔑Decisión — Mínimo Privilegio

Marcos te muestra la configuración del usuario app_user en MySQL. El sistema de login solo necesita leer y escribir en la tabla users. ¿Qué permisos debe tener?

-- Configuración actual (PELIGROSA):
GRANT ALL PRIVILEGES ON *.* TO 'app_user'@'localhost';

¿Cuál es la configuración correcta?

Intentos:

💡 Insight — Principio de Mínimo Privilegio

Cada componente del sistema debe tener solo los permisos que necesita para funcionar. Con GRANT ALL, si la app es comprometida, el atacante tiene control total. Con SELECT e INSERT solo en una tabla, el daño está contenido.

✅ Permisos corregidos. El jefe de seguridad entra a la sala: "¿En qué orden aplicamos los cambios? Necesitamos un plan."

💡 Insight — Orden de Remediación

Orden correcto: 1) Auditar logs (entender qué pasó), 2) Backup (nunca tocar producción sin respaldo), 3) Revocar privilegios (contener el ataque), 4) Probar en staging, 5) Aplicar en producción. El backup siempre antes de hacer cambios, para poder revertir si algo sale mal.
